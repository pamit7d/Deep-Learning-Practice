What is a BERT Tokenizer?

BERT (Bidirectional Encoder Representations from Transformers) is a model you learned in the Foundations of LLMs course. Unlike GPT-style models that just receive raw tokens, BERT requires its input to be formatted in a very specific way — with special structural tokens inserted at exact positions. The tokenizer is responsible for doing all of this automatically.

Core idea: BERT uses a WordPiece / BPE algorithm to split words into subword units, but on top of that, it wraps the output with special tokens like [CLS] and [SEP] that tell the model where a sentence starts, ends, and where two sentences meet.

Why does BERT need a specific input format?

Context
BERT was designed for two pre-training tasks:
1. Masked Language Modelling (MLM) — predict randomly hidden words in a sentence.
2. Next Sentence Prediction (NSP) — given two sentences, predict whether B follows A.

For NSP, the model needs to receive two sentences at once and know exactly where one ends and the other begins. That's why the [SEP] separator and the [CLS] classification token exist — they're baked into the input format.

Input Template (the BERT contract)

Single sentence
[CLS]token-A_1···token-A_n[SEP]
Sentence pair (A + B)
[CLS]token-A_1···token-A_n[SEP]token-B_1···token-B_m

Here A denotes sentence A with n tokens, B denotes sentence B with m tokens — as explained in the lecture.

Full Pipeline at a Glance

Raw textinput string
Normalizerlowercase
Pre-tokenizerwhitespace split
BPE Modelvocab=32k, ## prefix

Key concepts to keep in mind

WordPiece / BPE
Rare words get split into known subpieces. "hugging" → ['hug','##ging']. This keeps the vocabulary small while handling unseen words.
## (double hash)
BERT's signal that a subword is a continuation — it should be joined with the token before it. GPT-2 uses Ġ instead. Different models, different conventions.
Encoder vs Decoder
Here "encoder" = turning text into token IDs. "Decoder" = turning token IDs back into text. This is tokenizer encoding/decoding, NOT the transformer encoder/decoder architecture.
Post-processor
A configurable step that wraps tokens after BPE. It's separate from the BPE model so different models (BERT, RoBERTa, etc.) can share the same tokenizer code with different templates.

Special Tokens in BERT

These are reserved tokens added to the vocabulary before training begins. Their IDs are fixed — they're always the lowest-numbered entries because they're listed first in the trainer config.

ID assignment rule: The order you list tokens in special_tokens=[...] inside BpeTrainer determines their IDs — first in list = lowest ID.
["[PAD]","[UNK]","[CLS]","[SEP]","[MASK]"] → PAD=0, UNK=1, CLS=2, SEP=3, MASK=4
[CLS] — Classifier · ID: 2
Always the very first token in any BERT input. After the full transformer forward pass, only the hidden state at position 0 (the [CLS] position) is used as the "summary" of the whole sequence — fed into a classification head. It has no linguistic meaning; it's just a learnable slot for the model to accumulate sentence-level meaning.
[SEP] — Separator · ID: 3
Marks the end of a sentence or the boundary between two sentences in a pair. Combined with segment embeddings (type_id 0 for A, type_id 1 for B), it lets BERT know which tokens belong to which sentence — critical for NSP pre-training.
[PAD] — Padding · ID: 0
Training happens in batches. All sequences in a batch must be the same length. Shorter ones are padded with [PAD] to match the longest. The attention mask is set to 0 for pad positions so the model ignores them — padding is purely a batching convenience, not real input.
[MASK] — Masked · ID: 4
Used exclusively during MLM pre-training. About 15% of tokens are randomly replaced with [MASK], and the model must predict the original token. This is what makes BERT "bidirectional" — it sees context from both sides when predicting a masked token. At inference time, [MASK] is never used.
[UNK] — Unknown · ID: 1
A catch-all for any character sequence that cannot be represented even as a subword in the vocabulary. With a vocab size of 32,000 and aggressive subword splitting, this is rarely triggered — but it's always there as a safety net.
## — Subword Prefix (not a token)
Not a vocabulary token, but a prefix convention. When BPE splits "huggingface" into ['hug', '##ging', '##face'], the ## tells the decoder: "I am a continuation — strip my ## and join me to the previous token." This is how full words are reconstructed from pieces.

What a real tokenization looks like

From the lecture — encoding this sentence:
"All these are so simple to do in HF. Let's do more"
Token output
[CLS]allthesearesosimpletodoinh##f.let'sdomore[SEP]

Notice "HF" → h + ##f. After lowercasing, "hf" wasn't a single vocabulary entry, so BPE split it. Everything else was already in the vocabulary, so no further splitting was needed — which is why you don't see a ## on most tokens.

Sentence pair example (from the lecture)

Pair: sentence A + sentence B
[CLS] allthese···more [SEP] wehavealongwaytogo!

The second [SEP] at the very end is optional — some BERT variants add it, some don't. The key boundary is the [SEP] between A and B.

Building the BERT Tokenizer

You are assembling a tokenizer pipeline piece by piece. Think of it like connecting stages on an assembly line — each component does one job, and you can swap any component independently.

1
Create the base tokenizer with a BPE model
The Tokenizer object is the container. You hand it a BPE model which is the actual subword algorithm. The unk_token tells BPE what to emit when it encounters a completely unknown character.
from tokenizers import Tokenizer
from tokenizers.models import BPE

bert_tokenizer = Tokenizer(BPE(unk_token="[UNK]"))
2
Attach a Lowercase normalizer
Normalization runs before anything else. BERT-base is case-insensitive, so all input is lowercased first. This means "HuggingFace" and "huggingface" produce the same tokens — the model has half the vocabulary to learn.
from tokenizers import normalizers

bert_tokenizer.normalizer = normalizers.Lowercase()
Why normalise before tokenizing?
If you lowercase after BPE, "Apple" and "apple" would generate different subword merges during training, bloating the vocabulary unnecessarily. Normalizing first ensures the BPE algorithm always sees the same canonical form.
3
Attach a Whitespace pre-tokenizer
Pre-tokenization splits the normalized string into "words" before BPE runs. Whitespace splitting means BPE never merges across word boundaries — "do in" will never become one token. This makes the resulting vocabulary more interpretable.
from tokenizers.pre_tokenizers import Whitespace

bert_tokenizer.pre_tokenizer = Whitespace()
4
Create and configure the BPE trainer
The trainer holds the hyperparameters for learning the vocabulary from your corpus. It finds the most frequent pairs of characters/subwords and merges them repeatedly until the vocab reaches the target size.
from tokenizers.trainers import BpeTrainer

bert_trainer = BpeTrainer(
    vocab_size=32000,
    special_tokens=["[PAD]", "[UNK]", "[CLS]", "[SEP]", "[MASK]"],
    continuing_subword_prefix='##'
)
continuing_subword_prefix='##' — this tells the trainer: any subword that results from splitting a word in the middle should be prefixed with ##. So "playing" → ['play', '##ing']. Without this, the decoder wouldn't know which tokens to merge back into a word.
5
Train on your corpus
The trainer reads through your text files, counts character and subword pair frequencies, and builds the merge rules. After this, the tokenizer has a learned vocabulary specific to your data.
files = ["train.txt"]  # your text corpus files
bert_tokenizer.train(files, bert_trainer)

Post-processing — Inserting Special Tokens

At this point, the tokenizer can split text into subwords. But it doesn't yet know it should add [CLS] at the start and [SEP] at the end. That's what the post-processor does — it's a final transformation applied after BPE tokenization, before the IDs are returned to you.

Why is this a separate step? Because it's model-specific, not language-specific. The BPE vocabulary can be reused for many models, but each model has its own preferred input format. Keeping post-processing separate means you swap only this piece when changing models.

TemplateProcessing — how the template works

The template is a string that describes the desired output format using placeholders. $0 means "insert all tokens from the input here." For pairs, $A and $B are the two sentences.

single= template
"[CLS] $0 [SEP]"

For one sentence. $0 = all tokens from input. type_id defaults to 0 for everything.
pair= template
"[CLS] $A [SEP] $B:1"

For two sentences. $A = sentence A tokens (type_id 0). $B:1 = sentence B tokens (type_id 1). The :1 sets the segment ID.
from tokenizers.processors import TemplateProcessing

bert_tokenizer.post_processor = TemplateProcessing(
    single="[CLS] $0 [SEP]",
    pair="[CLS] $A [SEP] $B:1",
    special_tokens=[
        ("[CLS]", 2),  # string name + its vocabulary ID
        ("[SEP]", 3),  # must match IDs assigned during training
    ],
)
Why must we pass the IDs explicitly?
The post-processor runs after BPE — but it inserts tokens by their string names ("[CLS]"). It needs to know the corresponding integer ID to write into the output ids array. These must exactly match what BpeTrainer assigned (CLS=2, SEP=3).

type_id — the segment embedding signal

What type_id does
Each output token gets two things: its token ID (from the vocabulary) and a type_id (0 or 1). BERT adds a learned "segment embedding" for each type_id — this is how it knows, for a pair of sentences, which tokens belong to sentence A and which to sentence B. The :1 in $B:1 sets type_id=1 for all sentence B tokens. Everything without a suffix gets type_id=0.

Outputs from the lecture

Single sentence output
[CLS]alltheseare···more[SEP]
Sentence pair output
[CLS]all ··· more[SEP]wehavealongwaytogo!
Flexibility this gives you: If a future model wants [CLS] at the end rather than the beginning, you'd just write single="$0 [SEP] [CLS]". The tokenizer core stays unchanged — only this template changes.

Decoding — Token IDs Back to Text

After a model processes tokens and produces output IDs (e.g. in a generation or MLM task), you need to convert those IDs back into readable text. This is decoding. It's the reverse of encoding — but it has its own complications.

What decoding needs to do

1. Remove special tokens
Tokens like [PAD], [CLS], and [SEP] were added for the model's benefit. The end user shouldn't see them in the output. The decoder strips them automatically.
2. Merge ## subwords
BPE split "hf" into ['h', '##f']. The decoder needs to join them back. Without this step, every split word appears fragmented and unreadable.

The problem — decoding without a decoder

# IDs from the previous encoding step
plain_tokens = bert_tokenizer.decode(ids)
print(plain_tokens)
Output: all these are so simple to do in h ##f . let ' s do more we have a long way to go !

Special tokens ([CLS], [SEP]) are removed ✓ — but h ##f is not merged ✗. The ## is just printed as-is.

The fix — attach a WordPiece decoder

The WordPiece decoder knows the ## convention: any token starting with ## is a continuation and must be stripped of its prefix and concatenated to the previous token.

from tokenizers.decoders import WordPiece

bert_tokenizer.decoder = WordPiece(prefix='##')

plain_tokens = bert_tokenizer.decode(ids)
print(plain_tokens)
Output now: all these are so simple to do in hf. let ' s do more we have a long way to go!
Why is the decoder a separate configurable component?
Because different tokenizer families use different continuation signals. BERT uses ##. GPT-2 uses Ġ (a Unicode space character prefixed to non-initial pieces). SentencePiece uses ▁ to mark word starts. Keeping the decoder swappable means the same Tokenizer framework works for all of them — you just plug in the right decoder for your model.
Decoder must match the tokenizer type:
BERT / WordPiece → WordPiece(prefix='##')
GPT-2 / Byte-level BPE → ByteLevel()
SentencePiece → Metaspace()
Full list: HuggingFace Tokenizers Decoders docs

Putting it all together — encoder + decoder

Mental model
Think of the tokenizer as having two "sides." The encoder side (normalizer → pre-tokenizer → BPE model → post-processor) converts text to IDs. The decoder side converts IDs back to text. You configure both on the same Tokenizer object. Until you set a decoder, only the encoder side works properly. 
# — Encoder side (configured earlier) —
bert_tokenizer.normalizer     = normalizers.Lowercase()
bert_tokenizer.pre_tokenizer  = Whitespace()
bert_tokenizer.post_processor = TemplateProcessing(...)

# — Decoder side (new addition) —
bert_tokenizer.decoder = WordPiece(prefix='##')

# Now the full round-trip works correctly
ids   = bert_tokenizer.encode("Hello HuggingFace!").ids
clean = bert_tokenizer.decode(ids)
print(clean)  # → "hello huggingface!"

Full pipeline — all imports at once

from tokenizers import Tokenizer, normalizers
from tokenizers.models import BPE
from tokenizers.pre_tokenizers import Whitespace
from tokenizers.trainers import BpeTrainer
from tokenizers.processors import TemplateProcessing
from tokenizers.decoders import WordPiece
from pprint import pprint

# 1. Create tokenizer
bert_tokenizer = Tokenizer(BPE(unk_token="[UNK]"))

# 2. Normalizer + pre-tokenizer
bert_tokenizer.normalizer    = normalizers.Lowercase()
bert_tokenizer.pre_tokenizer = Whitespace()

# 3. Train
bert_trainer = BpeTrainer(
    vocab_size=32000,
    special_tokens=["[PAD]", "[UNK]", "[CLS]", "[SEP]", "[MASK]"],
    continuing_subword_prefix='##'
)
bert_tokenizer.train(["train.txt"], bert_trainer)

# 4. Post-processor
bert_tokenizer.post_processor = TemplateProcessing(
    single="[CLS] $0 [SEP]",
    pair="[CLS] $A [SEP] $B:1",
    special_tokens=[("[CLS]", 2), ("[SEP]", 3)],
)

# 5. Decoder
bert_tokenizer.decoder = WordPiece(prefix='##')

# 6. Use it
ids   = bert_tokenizer.encode("Hello HuggingFace!").ids
clean = bert_tokenizer.decode(ids)
print(clean)  # → "hello huggingface!"
One-line summary of each component:
normalizer → lowercase before anything else
pre_tokenizer → split on spaces so BPE never crosses word boundaries
BpeTrainer → learn subword merge rules; 32k vocab; ## for continuations
post_processor → wrap output with [CLS] and [SEP] per BERT's format
decoder → recognise ## prefix and merge subwords back into full tokens
