Part 1

Why SentencePiece? The Problem with BPE

BPE is greedy — once a merge is made, it's permanent. This means the final tokenization depends on merge order, not on what's actually best for the word.

What does "greedy" mean? At every step, BPE picks whatever looks best right now — like climbing a hill by always stepping on the nearest higher rock. You might reach a small hill instead of the mountain peak, because you never looked ahead.

Same vocabulary, different merge order → different result

Vocab: h, e, l, o, he, lo, ll, hello — all valid tokens

Merge order 1 → he merged first, then lo: hello → he + llo → he + l + lo Merge order 2 → ll merged first, then he: hello → h + e + ll + o → he + ll + o

⚠️ Same word, same vocabulary — only the merge order changed, yet the tokenization is different.

Part 2

Probability Model — How to Score a Segmentation

SentencePiece uses a Unigram Language Model: every token is treated as independent, so the probability of a segmentation is just the product of token probabilities.

What is "Unigram"? "Uni" = one. We look at each token alone, not in context of its neighbours. It assumes he appearing has nothing to do with whether ll appears next. This simplification makes the math clean and fast.

Why multiply probabilities?

Think of it like flipping coins. If P(Heads) = 0.5, then P(Heads AND Heads) = 0.5 × 0.5 = 0.25. Each event is independent, so we multiply. Same logic applies to tokens — we multiply because we assume each token is independently chosen.

Segmentation: he + ll + o Question: What's the chance we see all three in sequence? Answer: P(he) × P(ll) × P(o) ← because they're independent = 0.40 × 0.20 × 0.50 = 0.040

Why log probabilities? Multiplying many small decimals causes underflow.
log(a × b) = log(a) + log(b) — multiplication becomes addition, which is faster and stable.

Formula

Direct: P(X) = P(x₁) × P(x₂) × ... × P(xₙ) Log form: log P(X) = log P(x₁) + log P(x₂) + ... + log P(xₙ) Goal: X* = argmax P(X) ← find segmentation with max probability
Part 3

Training with the EM Algorithm

SentencePiece starts with a huge vocabulary and gradually prunes it — the opposite of BPE.

Latent Variable: The text corpus is observed, but the correct segmentation of each word is hidden. EM discovers it iteratively.

Why start large? Instead of guessing which 30K tokens are useful upfront, we collect everything (250K+) and let the algorithm delete the bad ones. It's easier to remove than to add — like writing every possible answer on a test and erasing the wrong ones.

E-Step — Estimate Probabilities

Look at the current best segmentation of every word. Count how often each token appears across the whole corpus. Then:

P(token) = freq(token) ───────────── total freq Example: Token "ing" appears 300 times Total all tokens = 920 P(ing) = 300/920 = 0.326

M-Step — Find Best Segmentation

Using the updated probabilities, re-tokenize every word by finding its highest-probability segmentation (via Viterbi). This may change which tokens get used.

Before M-step: "learning" → learn + ing After M-step (probs changed): "learning" → lea + rning Effect: freq(learn) ↓ freq(lea) ↑ freq(ing) ↓ freq(rning) ↑

Full Training Loop

Start: Large Vocab (e.g. 250K tokens)
↓
E-Step: Estimate token probabilities
↓
M-Step: Best segmentation per word (Viterbi)
↓
Update token frequencies & probabilities
↓
Remove bottom 20% lowest-prob tokens
↓
Repeat until target size reached (e.g. 30K)
Useless tokens like xyz, abcq are almost never selected in best segmentations → their probability approaches 0 → automatically removed.
Part 4

Viterbi Algorithm — Finding the Best Segmentation Efficiently

Brute-force search over all segmentations is exponential. Viterbi uses dynamic programming + pruning to find the optimal path in linear time.

Key insight: If path A beats path B at position i, then any extension of B can never beat the same extension of A. → Discard B immediately.

What is Dynamic Programming here?

Instead of recomputing from scratch for each segmentation, we store the best score up to each character position. When we move to the next character, we just look up the stored best — no recomputation needed.

Position 1: best[1] = best way to tokenize first 1 char Position 2: best[2] = best way to tokenize first 2 chars Position N: best[N] = answer! Each position only looks at: "What's the best split ending exactly here?" → Compare all tokens that END at this position → Keep only the winner, discard the rest

Example: tokenizing "WHERE" step by step

Vocab with log probs: W (−4.29), H (−3.34), WH (−5.99), E (−2.11), ER (−4.01), ERE (−4.50), WHERE (not in vocab). Higher = better (less negative).

WH WH −5.99 ✓ WH (−5.99) vs W+H (−4.29−3.34=−7.63) W+H = −7.63 ✗

Step-by-step reasoning

Position WH: We can either treat "WH" as a single token (−5.99) or as two separate tokens W+H (−4.29 + −3.34 = −7.63). Since −5.99 > −7.63, WH wins. The path W→H is pruned forever.

Position WHE: Only one valid option — extend the best path at WH by adding token E. Score = −5.99 + −2.11 = −8.10. Token "WHE" doesn't exist in vocab so that's ruled out.

Position WHER: Two options — extend from WHE by adding R (−8.10 + −3.56 = −11.66), OR extend from WH by adding ER as one token (−5.99 + −4.01 = −10.00). WH+ER wins because −10.00 > −11.66.

Position WHERE: Extend from WHER+E (−10.00 + −2.11 = −12.11) OR from WH+ERE as one token (−5.99 + −4.50 = −10.49). WH+ERE wins.

Pruning visualised

START │ ├─ W (−4.29) │ └─ W + H = −7.63 ✂ PRUNED (worse than WH) │ └─ WH (−5.99) ◀ BEST └─ WH + E (−8.10) └─ WH + ER (−10.00) ◀ BEST └─ WH + ERE (−10.49) ◀ FINAL ✓
Result: WHERE → WH + ERE with log score −10.49

Quick-fire interview answers

Q: Why is BPE greedy?
It always picks the most frequent pair right now, without checking future merges. Like always stepping on the nearest higher rock — you might end up on a small hill, not the mountain peak.

Q: Why use log probabilities?
Multiplying many small values (e.g. 0.4 × 0.2 × 0.1 × 0.05 × ...) quickly becomes 0.000000001 — computers lose precision. Log turns × into +, so we add clean negative numbers instead.

Q: What does Viterbi do?
Finds the highest-probability segmentation efficiently. At each character position, it stores only the best path reaching that point and discards all worse paths. This avoids checking every possible split combination.

Q: What is the EM algorithm's role?
E-step: Given the current segmentations, estimate how likely each token is (its probability). M-step: Given those probabilities, re-find the best segmentation for every word. The two steps feed each other — better probabilities → better segmentation → better probabilities — until convergence.

Q: Why does SentencePiece start with a large vocabulary?
Because it's hard to know upfront which 30K subwords will be useful. Starting with 250K and pruning weak ones is safer — tokens with near-zero probability are removed automatically since they never appear in optimal segmentations.
